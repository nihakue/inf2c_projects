Question 1

Why I picked django-cms

I have worked with two or three different cms packages in the past (concrete5, wordpress, and joomla), and I was always sort of agitated by them. When they weren't generating bloated, obfuscated html, they were filling my database with artifacts. To make matters worse, plugins and extensions for these projects are written primarily in php. I see django-cms as a powerful alternative to those tools. Django also claims to adhere to the DRY (Don't Repeat Yourself) and python PEP8 style guides, which is nice

I'm interested in exploring the inner workings of a cms, with the added benefit that as I exlpore, I will be improving my proficiency with python, picking up a new SDK (django), and getting experience with a tool that I will very likely use for my own web creations.

What is django-cms for?

django-cms helps developers create websites that are dynamic and maintanable by end-users and developers alike. It supervises database operations using python-mysql, provides a nice graphical front-end for content management, and acts as a platform for django plugins.

Getting it to run

I'm going to be honest... I picked a python project because I had this preconception that a python project would be easy to build. While this is still technically true, I ran into a number of issues when getting things up and running.

Hardware
White Macbook 2009 Dual Core with 8g of RAM
OS
MacOSX Lion
Python version
Python 2.6.6
Dependencies
  • Python 2.5 (or a higher release of 2.x).
  • Django 1.4.5, 1.5 or higher
  • South 0.7.2 or higher
  • PIL 1.1.6 or higher
  • django-classy-tags 0.3.4.1 or higher
  • django-mptt 0.5.2 (strict due to API compatibility issues)
  • django-sekizai 0.6.1 or higher
  • html5lib 0.90 or higher
  • django-i18nurls (if using django 1.3.X)
  • An installed and working instance of one of the databases listed in the Databases section.

Steps taken

Fork the django-cms repo on github
Also added an upstream remote, so that I can pull changes from the original repo
Clone the source to my mac
Discover that the version of python that ships with Mac is shit
Macintosh packages a version of python that is slightly outdated, and known to introduce bugs in some situation. Additionally, this python squashes all other isntallations of python, making it difficult to create python virtual environments.
Install homebrew (package management)
homebrew solves this problem by allowing you to install the official python distribution, and then overwrite linkages to the macosx python. I did some testing, and this doesn't appear to break other python projects
Use homebrew to install python 2.7.5 and pip
homebrew is a package distribution application for mac. Something that Mac desperatley lacks. Using homebrew, I install pip for python module management
Use pip to install virtualenv and virtualenvwrapper
virtualenv will allow me to test bugs accross versions of django-cms and all its dependencies. It also lets me go crazy installing python modules without having to worry about jacking something up.
pip install all the django-cms dependencies
Fortunately, django-cms devs provide a text file that I can use to recursively install all dependencies for a particular version of django-cms.
Run the end-to-end django-cms test
django-cms also provides a tes script that I can execute to make sure everything has been assembled correctly
Modify/run the project creation source script to create a custom new django-cms project (pictured)
After a little bit of exploring, I am able to modify the script responsible for creating new projects.

Question 2

django-cms hierarchy

django-cms is orginized, as many python projects, into a hierarchy of modules. The meat of the project resides in the cms module/directory, with supporting modules mostly sitting just above that in the django-cms module (with the exception of all utils). There are a number of important scripts in django-cms such as the setup script and testing.

Within the cms module, the core functionality is broken up into many submodules

Languages

It's worth noting that django-cms is as far as I can tell pure python. Django must be responsible for generating the html and javascript (although I haven't yet confirmed this)

Question 3

Documentation

django-cms uses Sphinx to maintain their documentation, and it seems to be working pretty well for them, but from what I have read on the IRC channel and in the google group for developers, new devs are having a hard time with outdated documentation. django-cms recently released a beta version 3, with some pretty drastic refactoring. Consequently, many of the tutorials have become outdated.

Starting Fresh

Currently, the method for creating a new django-cms projects involves a fairly hairy process of editing conf files. Granted django-cms is marketed as being strictly for devs, it still seems like it would be nice (and not to difficult) to make this process interactive and graphical.

The default/test project front page could potentially be extended to include a wizard that guides newcomers through the process of enabling installed plugins, setting certain directory variables, and then pass this information on the the users' configure.py using a json file and the already-in-place configure() method.

Internationalization

Another strong feature of django-cms is support for internationalization. Judging by the github issues, this is an area that is very often used, and still pretty buggy.
